# Configure Your Cluster

Cluster configuration lives in `config/clusters/<cluster-name>/`. You declare your cluster's shape once in a small set of human-authored YAML files; the render compiles them into all the generated artifacts the cluster needs. This guide explains what to configure and how the render cycle works.

For a detailed explanation of how the render works, see [Config Compiler](../concepts/config-compiler.md).

---

## Input Files

### `config/defaults/cluster.yaml` (read-only)

Template-owned defaults. **Do not edit this file.** It contains sensible provider-agnostic defaults for every field. You pull upstream template improvements here without conflicting with your cluster config.

### `config/clusters/<cluster-name>/cluster.yaml` (user-owned)

Your cluster-specific overrides. Only the values that differ from the defaults need to appear here. The render deep-merges this over the defaults (your values win per-key).

### `config/clusters/<cluster-name>/nodes.yaml` (user-owned, optional)

Per-node hardware configuration: node identity, network interface selectors, and disk selectors for nodes with heterogeneous hardware. If all nodes share the same hardware profile, this file can be left as `nodes: []`.

### `config/clusters/<cluster-name>/talos/` (user-owned)

Per-cluster Talos patch overlays applied on top of the template-owned base configs. See [Talos Overlays](#talos-overlays) below.

## Key Configuration Areas

All fields are documented in the [cluster.yaml schema reference](../reference/cluster-yaml-schema.md). The most common settings are:

### Cluster Identity and Network

```yaml
cluster:
  name: <cluster-name>            # short identifier; used in node hostnames
  kubernetesVersion: v1.32.0      # Kubernetes version to deploy
  vip: <ip-address>               # floating IP for the Kubernetes/Talos API
  # endpoint is computed from vip by default: https://<vip>:6443
  # Override only if the API is reached through a different address.
  certSANs:
    talos: []                     # extra IPs/hostnames for Talos API TLS certs
    kubernetes: []                # extra IPs/hostnames for kube-apiserver TLS certs
```

### Provider

```yaml
provider:
  name: hcloud    # hcloud | manual
```

This determines which provider inventory and group_vars are loaded, and which provider-gated features are available.

### Talos Version and Extensions

```yaml
talos:
  version: v1.13.3
  extensions:
    base: []      # extensions included in every image (e.g. qemu-guest-agent)
    extra: []     # additional user-specified extensions
  network:
    defaultInterface:
      # Prefer deviceSelector (stable hardware identity) over interface names.
      deviceSelector:
        hardwareAddr: '<mac-prefix>:*'
      # Or fall back to interface name:
      # interface: eth0
```

### CNI

```yaml
cni:
  name: cilium    # cilium | none
```

`cilium` disables Talos's built-in CNI management and installs Cilium as a pre-Flux component. `none` leaves in-cluster CNI management active.

### Cloud Controller Manager

```yaml
cloud_controller_manager:
  enabled: auto   # auto | true | false
```

`auto` resolves to `true` for providers with a supported CCM (currently `hcloud`) and `false` for `manual`. Use `false` to opt out on cloud providers; use `true` to force-enable (validation fails if the provider has no supported CCM).

### Features

Feature flags control which add-on controllers and services Flux deploys after bootstrap:

```yaml
features:
  certManager: false
  ingress: none       # none | envoy-gateway | traefik (pick one)
  openebs: false
  hcloudCsi: false    # provider-gated: requires provider.name: hcloud
```

> **Constraints enforced at render time:**
> - `ingress` is pick-one — enabling two ingress controllers is a validation error.
> - Features that need TLS certificates require `certManager: true`.
> - Provider-gated features (`hcloudCsi`) are rejected if `provider.name: manual`.

### Storage Volumes

```yaml
storage:
  volumes:
    - name: <volume-name>
      type: raw
      diskSelector: system_disk
      minSize: 20GB
      maxSize: 20GB
      openebs: true   # mark as OpenEBS LVM backing device
```

User-owned lists replace the default entirely (they are not merged). The schema rejects encryption keys (`additionalProperties: false`) due to a known Talos LUKS+LVM issue.

### Flux

```yaml
flux:
  provider: generic               # generic | github | bitbucket-server
  repoUrl: 'ssh://git@<host>/<owner>/<repo>.git'
  repoBranch: main
  path: ./flux/clusters/<cluster-name>
  auth:
    mode: ssh-private-key         # ssh-private-key | https-token | provider-deploy-key
    # ... additional auth fields depending on mode
```

See `config/defaults/cluster.yaml` and [ADR-015](../adr/0015-flux-bootstrap-auth.md) for all auth options.

## The Render Cycle

After editing `cluster.yaml` or `nodes.yaml`, run the render playbook from `infra/ansible/` to compile your inputs:

```shell
ansible-playbook playbooks/render-config.yml -e config_cluster=<cluster-name>
```

The render:
- Deep-merges `config/defaults/cluster.yaml` with `config/clusters/<cluster-name>/cluster.yaml`
- Validates the merged config against `config/schemas/cluster-input.schema.json`
- Emits generated `group_vars/`, Talos machine-config patches, and Flux kustomization selections
- Composes Talos patches from `config/talos/base/`, `config/talos/components/`, and `config/clusters/<cluster-name>/talos/`

Generated files are written in-place and marked *do not edit*. Commit them before running live playbooks.

### Drift Detection

To check whether the committed generated files are up-to-date without writing anything:

```shell
ansible-playbook playbooks/render-config.yml \
  -e config_cluster=<cluster-name> \
  -e config_render_check=true
```

This renders to a temp directory and diffs against the committed tree. Any uncommitted hand-edit or stale generated file is reported as drift and fails CI. See [Render and Drift](../contributing/render-and-drift.md) for details.

### When to Re-Run the Render

| Change | Render required | `bootstrap-talos.yml` required |
|---|---|---|
| Enable/disable a feature | Yes | Only for Talos-coupled features |
| Kubernetes or Talos version | Yes | Yes |
| CNI or CCM settings | Yes | Yes |
| Talos extensions | Yes | Yes (upgrade-talos or re-bootstrap) |
| Network/disk/volume config | Yes | Yes |
| Flux settings only | Yes | No |
| Kustomize patch in Flux overlay | No | No |
| Add or update an app | No | No |

## Talos Overlays

The cluster-level Talos overlay directory at `config/clusters/<cluster-name>/talos/` contains user-authored Talos patch manifests applied on top of the template-owned base configs:

```
config/clusters/<cluster-name>/talos/
  all.yaml          # patches applied to every node
  controlplane.yaml # patches applied to controlplane + hybrid nodes only
  worker.yaml       # patches applied to worker + hybrid nodes only
  nodes/            # per-node patches (for heterogeneous hardware)
```

These are standard Talos strategic-merge patches. The render composes them with the template-owned base configs and component fragments from `config/talos/` in a deterministic order.

**When to add a custom overlay:**

- Customizing kubelet arguments
- Adding node labels or taints not covered by the template
- Hardware-specific network or disk configuration that cannot be expressed in `cluster.yaml`

For configuration that fits the `cluster.yaml` schema (extensions, VIP, Kubernetes version, storage volumes), prefer the schema — it benefits from validation and template maintenance. Use raw Talos overlays only for settings outside the schema's scope.

See [Layered Talos Config](../concepts/layered-talos-config.md) for the full patch merge order.
