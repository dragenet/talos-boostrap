# Node roles

> **All `group_vars/` files referenced below are render-generated by
> `playbooks/render-config.yml` from the selected cluster's inputs under
> `config/clusters/<cluster>/`.** Specifically: `cluster.yaml` (provider/features/versions),
> `nodes.yaml` (node list/roles), and the Talos-shaped patch overlays under
> `config/clusters/<cluster>/talos/*` (per-role and per-node). Committed generated files
> start with `# Generated by render-config; do not edit.` Edit the cluster inputs
> and re-render — never hand-edit the generated files.

## The three roles

| Role | Short code | Runs etcd | Schedules workloads | `allowSchedulingOnControlPlanes` |
|---|---|---|---|---|
| `controlplane` | `cp` | yes | no (taint) | false |
| `worker` | `wk` | no | yes | n/a |
| `hybrid` | `hb` | yes | yes | true |

**Why hybrid?** A 3-node cluster that is pure controlplane has no worker capacity. A
3-node cluster of pure workers has no Kubernetes API. Hybrid nodes do both — all three
nodes participate in etcd and all three accept workload pods. The trade-off is blast
radius: a bad deployment can saturate resources that etcd needs. For a homelab this is
acceptable; production clusters typically separate the roles once the node count permits.

## Naming convention

Hostnames follow `{cluster_name}-{short_code}{ordinal}`, e.g. `example-hb1`, `example-hb2`,
`example-hb3`. The short code is the only place abbreviations appear. Everywhere else —
Ansible group names, Hetzner server labels, Kubernetes node labels — the full name is
used:
- Ansible group: `hybrid`, `controlplane`, `worker`
- Hetzner label: `role: hybrid`
- Kubernetes label: `node-role.kubernetes.io/hybrid: ""`

The short code map lives in `inventories/common/group_vars/all/cluster.yml` (render-generated from
`config/clusters/<cluster>/cluster.yaml` `cluster.nodeRoleShort`):
```yaml
node_role_short:
  controlplane: cp
  hybrid: hb
  worker: wk
```

## How Kubernetes sees the role label

Kubernetes normally prevents non-system components from setting
`node-role.kubernetes.io/*` labels because they're in the restricted prefix and kubelet
ignores them. Talos applies `machine.nodeLabels` via its own node controller (which runs
as a system component), bypassing that restriction. The label lands on the Node object
after the node registers.

The `allowSchedulingOnControlPlanes: true` patch (hybrid nodes only) removes the
`node-role.kubernetes.io/control-plane:NoSchedule` taint that Talos sets by default on
controlplane nodes. Without it, the Kubernetes scheduler skips those nodes for regular
pods.

## Changing the topology

Topology is declared in `inventories/providers/hcloud/group_vars/all/hcloud.yml` (render-generated from
`config/clusters/<cluster>/cluster.yaml` `hcloud.topology`). Per-node role assignment lives in
`config/clusters/<cluster>/nodes.yaml`:
```yaml
hcloud_topology:
  controlplane: 0
  hybrid: 3
  worker: 0
```

To change counts:
1. Edit `hcloud.topology` in `config/clusters/<cluster>/cluster.yaml` (and/or per-node
   `node_role` entries in `config/clusters/<cluster>/nodes.yaml`).
2. Re-render: `ansible-playbook playbooks/render-config.yml`.
3. Re-run `provision-infra.yml` to create or delete servers to match.
4. The Hetzner dynamic inventory plugin then picks up the new servers automatically (it
   queries by `cluster=example` label). Run `bootstrap-talos.yml` afterward to apply configs
   to any new nodes.
