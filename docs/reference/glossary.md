# Glossary

Definitions for terms used throughout the documentation. Terms are listed alphabetically.

---

**Bootstrap**
The initial one-time sequence that brings a cluster from zero to a fully running, GitOps-reconciled state. Involves provisioning infrastructure (`provision-infra.yml`), applying Talos machine configs (`bootstrap-talos.yml`), and installing Flux (`bootstrap-flux.yml`). Once completed, day-to-day cluster management is handled by Flux without re-running the bootstrap playbooks.

---

**Config compiler**
The `render-config.yml` playbook (and its underlying `config_render` Ansible role). It reads `config/defaults/cluster.yaml` and `config/clusters/<cluster>/cluster.yaml`, deep-merges them, validates the result against the JSON schema, and emits generated artifacts: Ansible `group_vars` and Talos machine-config patches. The compiler is the single writer for all generated files; no other process or human should edit them directly. See ADR-011.

---

**GitOps**
Using a Git repository as the single source of truth for cluster state, with an in-cluster agent (Flux) continuously reconciling the live cluster to match what is declared in git. Changes are made by committing to git; the cluster converges automatically. This template uses [FluxCD](https://fluxcd.io) as the GitOps engine.

---

**HelmRelease**
A FluxCD custom resource (`HelmRelease`) that manages the lifecycle of a Helm chart deployment — install, upgrade, and rollback — declaratively from git. Flux watches HelmRelease objects and reconciles the corresponding Helm release in the cluster.

---

**Kustomization**
In the context of this repo, a FluxCD `Kustomization` custom resource that references a path of YAML manifests in git and instructs Flux to apply them to the cluster. Not to be confused with `kustomize` (the tool) or a `kustomization.yaml` file, though Flux Kustomizations can use kustomize overlays under the hood.

---

**Machine config**
A Talos-specific YAML configuration file for a single node, generated by `talosctl gen config` and customized with patches. It encodes the node's network settings, disk configuration, Kubernetes role, API endpoints, and system extensions. Machine configs are applied to nodes via the Talos maintenance API during bootstrap. In this template, per-role configs (`controlplane.yaml`, `worker.yaml`) are generated by the `talos_config` role.

---

**Node role**
An assignment that determines both the Talos machine type and the Kubernetes function of a node. The supported roles are:

- `controlplane` — etcd + Kubernetes control plane only, no workloads
- `hybrid` — control plane + workloads on the same node
- `worker` — workloads only, no etcd or control-plane components

Node roles are declared in the inventory via the `node_role` hostvar and correspond directly to Talos machine types (`controlplane` → Talos controlplane, `worker` → Talos worker). See the [Inventory Contract](inventory-contract.md).

---

**Overlay**
A Talos config patch file that is merged into the base machine config at render time. Overlays allow cluster-specific or node-role-specific customizations (network settings, disk mounts, system extensions, etc.) without modifying the shared base templates. In this repo, overlays live under `config/talos/base/`, `config/talos/components/`, and `config/clusters/<cluster>/talos/`, applied in priority order by the config compiler.

---

**Provider**
The infrastructure layer that supplies compute nodes for the cluster. Providers are abstracted behind a standard [inventory contract](inventory-contract.md). This template ships with two providers:

- `hcloud` — Hetzner Cloud (automated provisioning via Ansible)
- `manual` — bare-metal or any other cloud (operator pre-provisions nodes; static inventory)

Provider-specific logic lives in `infra/ansible/playbooks/providers/` and `infra/ansible/roles/providers/`.

---

**Talos Linux**
An immutable, minimal, API-driven Linux operating system purpose-built for running Kubernetes nodes. Talos has no SSH, no shell, and no package manager — all configuration and management is done via the Talos API (`talosctl`). Its immutability and controlled surface make it well-suited for reproducible, GitOps-managed clusters. See [talos.dev](https://www.talos.dev).
