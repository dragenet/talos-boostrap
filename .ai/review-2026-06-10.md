# kube1 Repository Review — 2026-06-10

Reviewed against the repo's own `CLAUDE.md` standards and general IaC / Talos / Flux / Ansible best practice. Findings are grounded in `ansible-lint` output and `git` state, not assumptions.

## TL;DR

The Ansible **roles** (`hcloud_servers`, `talos_config`) are genuinely high quality — idempotent, namespaced, and commented with the *why* exactly as `CLAUDE.md` demands. The weak spot is `create_talos_image.yml`, which is **not held to the same bar** and currently makes the tree **fail `ansible-lint`** (4 failures). Secrets handling is correct at the git boundary (everything sensitive is gitignored), but the live Hetzner token sits in plaintext on disk and the repo mandates SOPS/Vault. There is no automation to enforce any of the standards `CLAUDE.md` lays out.

Overall: **solid foundations, a few sharp edges, and a gap between the rules written down and the rules enforced.**

Separately, two forward-looking goals were raised: (1) evolve the Ansible toward a **provider-portable** layout (future non-Hetzner clouds + manually-provisioned machines) by renaming playbooks per concern and splitting vars; and (2) revisit the **node public-IP discovery/storage**, which is suspected of not being idiomatic. Both are addressed in [Architecture direction: provider portability](#architecture-direction-provider-portability-requested) below, and tracked as findings **M6 / M7**.

## What's done well (keep doing this)

- **`module_defaults` for the API token** (`group/hetzner.hcloud.all` → `api_token`) — never inlined, exactly per policy.
- **Variable namespacing** is followed precisely: `hcloud_servers_*` / `talos_config_*` for role-local, `hcloud_` / `talos_` / `cluster_` / `flux_` for global.
- **Idempotency in the roles** is real, not cosmetic: `creates:`, `changed_when`, `until/retries/delay`, and the maintenance-mode detection in `talos_config` (distinguishing "unconfigured node" from "transient apid timeout") is the standout — well reasoned and well commented.
- **The etcd member-IP comment** (public-IP-to-`--control-plane-nodes` failure) and the **spread placement group rationale** are model examples of "comment the why."
- **Secrets are correctly gitignored.** `git status --ignored` confirms `secrets.yml`, `node_ips.yml`, `talos/secrets/`, and `talos_snapshot_id` are all ignored; the `secrets/` dir is `0700`. Nothing sensitive is staged.
- **`CLAUDE.md`'s "implementation status" is honest** — `flux_bootstrap` is a real stub, the Flux kustomizations are real `resources: []` placeholders, and `flux_repo_url` is genuinely unset. No vaporware described as running.
- **Flux ordering is correct**: `dependsOn` chains controllers → configs → apps, `prune: true` everywhere, `wait: true` on infra layers, intervals pinned.

## Findings by severity

### 🔴 Critical

**C1 — `ansible-lint` fails; the tree violates its own "lint must pass" rule.**
`CLAUDE.md` states "run `ansible-lint` then `--check --diff`" before handoff, but the tree currently reports **4 failures + 4 warnings**, all in `create_talos_image.yml`. The most dangerous is:

- **`risky-shell-pipe` (line 98).** The image build is `curl … | xz -dc | dd of=/dev/sda` with no `set -o pipefail`. If `curl` fails or the download truncates, the pipeline's exit status comes from `dd`, which happily writes whatever bytes it got and **returns success**. Result: a corrupt or empty Talos snapshot that the playbook reports as built — and this is the *very first* thing you run, so the failure surfaces much later as nodes that won't boot. **Fix:** `set -o pipefail` (the task already sets `executable: /bin/bash`, so this is one line), and verify the downloaded size / `dd` byte count.

**C2 — Live Hetzner API token in plaintext, and now exposed.**
`secrets.yml` holds a real, full-scope Hetzner Cloud token (server create/delete, snapshots, firewall). It is gitignored (good), but:
- The repo's **own Secrets policy** says "Use Ansible Vault or SOPS for any secrets that must live in the repo." This file lives in the repo tree (just untracked); SOPS-encrypting it would let it be committed safely and survive a machine loss.
- This token has now been read during review. **Rotate it** in the Hetzner console and re-store the new one (ideally SOPS-encrypted). A leaked token = full control of the project's infra.

### 🟠 High

**H1 — `create_talos_image.yml` isn't held to the repo's engineering standards.** The roles are exemplary; this playbook is the outlier:
- `command-instead-of-module` + `no-changed-when` on the `dd` task — needs a `changed_when:` and a `# noqa` with a reason (a streaming `curl|dd` genuinely can't use `get_url`, so document that).
- Four `args[module]: missing required arguments: api_token` warnings — the `hcloud_servers` role already silences these with `# noqa: args[module]` and an explanatory comment; the same hcloud tasks here lack it. Inconsistent.
- `risky-file-permissions` (line 148) — the snapshot-ID `copy` has no `mode:`; set one (e.g. `0644`).

**H2 — Firewall lock-out footgun via auto-detected admin IP.**
`hcloud_admin_ips` defaults to `[]`, which resolves to *"my current public IP"* via `ifconfig.me`. Consequences:
- A dynamic/residential IP change silently invalidates the firewall rules for Talos API (`50000`) and K8s API (`6443`) → you lose `talosctl`/`kubectl` until you re-run `provision.yml` or edit the firewall by hand. On Talos (no SSH) there's no side door.
- A single third-party (`ifconfig.me`) is a hard dependency for a security-critical value. Consider documenting this loudly, allowing an explicit list as the norm, and/or pinning a stable egress IP. This pairs with the eventual Cloudflare Tunnel plan (which would remove the need to expose `6443` at all).

### 🟡 Medium

**M1 — `requirements.yml` uses `version: ">=6.0.0"`, contradicting "Pin versions."**
`>=` is a floor, not a pin — a future `hetzner.hcloud` 7.x with breaking changes would be pulled in. `CLAUDE.md` explicitly says "Pin versions: collections in `requirements.yml`." Pin exactly (`6.x.y`) or use compatible-release (`~=6.0`) at minimum.

**M2 — Intra-cluster firewall is broader than the subnet.**
The "All TCP/UDP intra-cluster" rules allow `10.0.0.0/16` (`hcloud_private_network_cidr`), but nodes live in the `10.0.0.0/24` subnet. Scope these to `hcloud_private_subnet_cidr` to shrink blast radius if the network is ever extended with other subnets.

**M3 — `ansible.cfg` uses the deprecated `collections_paths` and a global path.**
Use the singular `collections_path`. More importantly, it points at `~/.ansible/collections`, so the pinned collection isn't installed deterministically per-project — a stale global install would mask the pin in `requirements.yml`. Consider a project-local collections path so "pinned" is actually what runs.

**M4 — Rendered secret-bearing files are world-readable (`0644`).**
`controlplane.yaml` (embeds cluster CA + keys), `talosconfig`, and `worker.yaml` are `0644`; only the parent dir's `0700` protects them. Defense-in-depth: have the role `chmod`/`file` them to `0600` after `talosctl gen config`. (`talosctl` writes at the default umask, so this needs an explicit step.)

**M5 — Empty `docs/` and no README.**
`CLAUDE.md` describes `docs/` as the home for "architecture decisions and runbooks," but it's just a `.gitkeep`. All living knowledge (architecture, status, workflows) is in `CLAUDE.md`, which is an *agent* instruction file. A human landing on the repo has no README. At minimum, a short README pointing at `CLAUDE.md` + an ADR or two would help.

**M6 — Node public-IP discovery uses a hand-written file as cross-play IPC; not idiomatic, not portable.** *(your call-out #2)*
`hcloud_servers` writes `infra/talos/node_ips.yml` and `talos_config` reads it back via `include_vars`. It works, but it uses the **filesystem as a side channel** between two separate `ansible-playbook` invocations, and the file contract (`node_public_ips` keyed by inventory name) is Hetzner-shaped — a manual or other-cloud backend would have to *fabricate* this file. The idiomatic Ansible answer is **dynamic inventory** (verified: `hetzner.hcloud.hcloud`), which queries the API at runtime and exposes `hcloud_ipv4` / `hcloud_private_ipv4` / `hcloud_labels`. Full treatment in [Architecture direction](#architecture-direction-provider-portability-requested) §D.

**M7 — Structure is Hetzner-coupled at the playbook + vars layer.** *(your call-out #1)*
Playbook names (`provision.yml`, `bootstrap.yml`) don't say *what* or *for which provider*, and `group_vars/all/vars.yml` mixes Hetzner-specific, Talos, and provider-agnostic cluster vars in one file. This is fine for one cloud but blocks the "add a second provider / manual nodes" goal. Recommended renaming + vars split in [Architecture direction](#architecture-direction-provider-portability-requested) §B–C. (The *roles* are already well-separated — `hcloud_servers` is provider-named, `talos_config` is provider-agnostic — so the refactor is mostly at the orchestration/vars layer, not the roles.)

### 🟢 Low / hygiene

- **L1 — No CI / pre-commit to enforce the mandated checks.** `CLAUDE.md` requires `ansible-lint` + `--check --diff`, yet nothing runs them automatically — and the tree is currently red (C1). A pre-commit hook + a GitHub Action would make the standard real instead of aspirational.
- **L2 — `worker.yaml` is generated but never used** (all nodes are control-plane). Harmless (gitignored), but dead output; `talosctl gen config` could be scoped, or just note it.
- **L3 — `.gitignore` doesn't cover tool/output dirs.** `graphify-out/`, `.opencode/`, and the nested `infra/ansible/.opencode/` are untracked but un-ignored — a `git add .` would commit them. Decide intentionally: are graph artifacts meant to be versioned? (The nested `.opencode/` under `infra/ansible/` also looks misplaced.)
- **L4 — `create_talos_image.yml` second run isn't a no-op.** It re-arms rescue, resets, and creates a *new* snapshot rather than detecting the existing `talos_snapshot_id`. It's documented "run once," so acceptable, but slightly at odds with the "idempotency is mandatory" line; a `creates:`-style guard on `talos_snapshot_id` would close the gap.
- **L5 — No LICENSE.** Fine for a private homelab; flag only if it'll ever be shared.
- **L6 — Verify Talos ↔ Kubernetes version support.** `talos_version: v1.13.3` with `kubernetes_version: v1.36.0` — confirm `v1.36.0` is within Talos 1.13's supported k8s range (check the Talos support matrix; don't assume).

## Architecture direction: provider portability (requested)

Goal: keep the Ansible usable when nodes come from **a different cloud** or are **provisioned manually**, without rewriting the Talos/Flux layers each time. The guiding principle is a clean **seam**: the *provisioning* layer (provider-specific — "make machines exist, networked, reachable") produces a small, stable **contract**; the *configuration* layer (Talos → Flux, provider-agnostic — "turn machines into a cluster") consumes only that contract and never names a provider.

The good news: your **roles already sit on the right side of that seam**. `hcloud_servers` is the only Hetzner-aware role; `talos_config` and `flux_bootstrap` are already provider-neutral *except* for how they receive node IPs (§D). So this is mostly an orchestration + vars reshuffle, not a rewrite.

### The contract (what every provider must produce)

Per node, the bootstrap layer needs exactly four things:

| Contract field | Hetzner source | Manual source |
|---|---|---|
| Reachable Talos-API address | public IPv4 (`hcloud_ipv4`) | static `ansible_host` in `hosts.yml` |
| Private IP (VIP / etcd) | `hcloud_private_ipv4` | static `node_private_ip` in `hosts.yml` |
| Role (control-plane / worker) | server label → group | inventory group membership |
| Cluster identity | shared `group_vars/all` | shared `group_vars/all` |

Standardize on **hostvar names** for the first three (e.g. `node_public_ip`, `node_private_ip`, and group membership for role). Any backend just has to populate those — Hetzner via dynamic inventory, manual via static inventory. `talos_config` reads `hostvars[...]` and stays ignorant of the source.

### A. Layering

```
provision-<provider>.yml   → provider-specific role (hcloud_servers, …)   ─┐
                                                                            ├─ produces the contract (inventory hostvars)
(manual)                   → static hosts.yml                             ─┘
        │
        ▼
bootstrap-talos.yml        → talos_config        (provider-agnostic)
bootstrap-flux.yml         → flux_bootstrap       (provider-agnostic)
```

### B. Playbook renaming (your call-out #1)

| Now | Proposed | Why |
|---|---|---|
| `create_talos_image.yml` | `image-talos-hetzner.yml` | Talos image build via Hetzner rescue+snapshot is **Hetzner-specific** — the name should say so. |
| `provision.yml` | `provision-hetzner.yml` | Frees `provision-<other>.yml` later; a manual setup has no provision step. |
| `bootstrap.yml` | `bootstrap-talos.yml` | Provider-agnostic; names the *concern* (Talos/etcd), not the cloud. |
| `addons.yml` | `bootstrap-flux.yml` | Same — names the concern; "addons" is vague. |

Naming convention to adopt: `<verb>-<concern>[-<provider>].yml`. Provider suffix **only** when the playbook is genuinely provider-bound. (Alternative if the list grows: group by directory — `playbooks/hetzner/…` vs `playbooks/talos/…` — but flat hyphenated names are fine at this size. Update the `## Key workflows` block in `CLAUDE.md` in the same change.)

### C. Vars split (your call-out #1)

Ansible auto-loads **every** file in a `group_vars/<group>/` directory, so splitting is free — no include wiring. Move from one `group_vars/all/vars.yml` to:

```
group_vars/
  all/
    cluster.yml   # provider-agnostic: cluster_name, cluster_vip, cluster_endpoint, kubernetes_version
    talos.yml     # talos_version, talos_schematic_id      (the OS layer, any provider)
    flux.yml      # flux_repo_url, flux_repo_branch, flux_path
  hetzner.yml     # hcloud_* : location, server_type, talos_image (snapshot id),
                  #            private_network_*, firewall_name, ssh_key_name, admin_ips
```

Then make the nodes members of a **`hetzner` group** in `hosts.yml` (alongside `controlplane`). Provider vars attach to the group, so a future `manual` group with its own `group_vars/manual.yml` (or per-host `host_vars/`) slots in without touching `all/`. Note `hcloud_talos_image` (a Hetzner *snapshot ID*) belongs in `hetzner.yml`, not `talos.yml` — it's the provider's artifact, distinct from `talos_version`/`talos_schematic_id` which are provider-neutral. Keeps the existing `hcloud_`/`talos_`/`cluster_`/`flux_` prefixes, just files them by layer.

### D. Node IP discovery → dynamic inventory (your call-out #2)

Replace the `node_ips.yml` write/read with the **`hetzner.hcloud.hcloud` dynamic inventory plugin** (FQCN and behavior verified against current docs). Sketch:

- In `provision-hetzner.yml`, **label** each server (e.g. `cluster=kube1`, `role=controlplane`) instead of writing a file.
- Add `infra/ansible/inventory/hcloud.yml` (the filename **must** end in `hcloud.yml`/`.yaml`):
  ```yaml
  plugin: hetzner.hcloud.hcloud
  token: "{{ lookup('ansible.builtin.env', 'HCLOUD_TOKEN') }}"   # or via secrets
  label_selector: cluster=kube1
  keyed_groups:
    - key: hcloud_labels.role        # builds a 'controlplane' group from the label
      prefix: ""
      separator: ""
  compose:
    node_public_ip: hcloud_ipv4
    node_private_ip: hcloud_private_ipv4
  ```
- `bootstrap-talos.yml` runs against this inventory; `talos_config` reads `hostvars[item].node_public_ip` / `.node_private_ip`. **Delete** the "Write node public IPs" copy task, the `node_ips.yml` defaults, and the `include_vars`.

Why this is better:
- **Idiomatic** — Ansible's built-in answer to "current infra state" is dynamic inventory, not a hand-rolled file passed between runs.
- **Portable** — manual nodes use the same `node_public_ip` / `node_private_ip` hostvars from static `hosts.yml`; `talos_config` doesn't change. The seam from §A becomes real.
- **No stale state** — IPs are read live, so a recreated server can't desync a file on disk.

Trade-offs / watch-outs:
- The plugin hits the Hetzner API on every run and needs the token available to **inventory loading** (env `HCLOUD_TOKEN` or the config `token:`), which is a different code path than the playbook `module_defaults` — wire both, and keep the token out of git (ties to **C2**: SOPS or env).
- Chicken-and-egg: dynamic inventory only sees servers that already exist, so it's for `bootstrap-*`, not the `provision`/`image` steps (those still run against `localhost`). That's the natural split anyway.
- **Private IPs**: today they're *assigned* statically in `hosts.yml` (desired state) and reused in the Talos config. The plugin reports the *actual* assigned `hcloud_private_ipv4` (discovered state) — fine to consume for config, but keep the static assignment as the source of truth at attach time so VIP/etcd addressing stays deterministic.
