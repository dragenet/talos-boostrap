# Render/config-compiler refactor ‚Äî closing report

**Date:** 2026-06-22
**Plan:** `.ai/plans/2026-06-22-refactor-render-architecture.md`
**Type:** Pure refactor ‚Äî no behavior change, no live cluster operations
**Closes:** ADR-010 / 011 / 012 / 013 build-out (the architecture that was already
documented but not implemented); the prior provider-portability refactor
(`.ai/reports/2026-06-22-provider-portability-refactor.md`) closed the inventory
half (ADR-014) and is referenced, not reworked.

## What changed

The hand-authored ADR-007-era inputs (per-node group_vars, two `*.j2` Talos
patches, empty Flux placeholders) were migrated into a single render-driven
pipeline. The compiler is the new source of truth; humans edit overrides, the
render emits the rest.

### Config schema (ADR-011)

- **New** `config/defaults/cluster.yaml` ‚Äî template-owned, provider-agnostic
  defaults with schema-shaped comments: cluster identity, Talos install image,
  `talos.network.defaultInterface.interface: eth1`, `features.cni: cilium`,
  `features.{certManager,ingress,openebs,hcloudCcm,hcloudCsi}: false` (all
  off), empty `storage.volumes`.
- **New** `config/overrides/cluster.yaml` ‚Äî user-owned kube1-specific choices
  migrated verbatim from the prior hand-authored group_vars: `cluster.name:
  kube1`, `vip: 10.0.0.10`, `kubernetesVersion: v1.36.0`, `talos.version:
  v1.13.3`, `provider.name: hcloud`, hcloud topology (3 hybrid nodes, nbg1,
  server-type fallback lists, network/firewall names, IP offset 11), empty
  `certSANs`, etc.
- **New** `config/overrides/nodes.yaml` ‚Äî user-owned per-node hardware
  selectors and raw node patches (empty for now; structure in place).

### Render compiler (ADR-011)

- **New** `infra/ansible/playbooks/render-config.yml` ‚Äî the single entrypoint.
- **New** `infra/ansible/roles/config_render/` with the planned task split:
  `load ‚Üí validate ‚Üí compute ‚Üí render_ansible / render_talos / render_flux ‚Üí
  check`. The compiler deep-merges `defaults/` over `overrides/` (overrides
  win per-key), validates the merged tree, computes derived values
  (`talos_extensions`, `bootstrap_pre_flux_components`, the
  `externalCloudProvider` gate), and renders to three target trees.
- **Computed-key guard:** directly overriding a computed key (e.g. final
  `talos_extensions`) is rejected with a clear error ‚Äî keeps one writer
  per file.
- **Render check mode:** `-e config_render_check=true` or `--tags check`
  renders to a temp dir and diffs against the committed generated tree. No
  CI yet (per plan: local-only until CI is introduced later).

### Layered Talos patches (ADR-013)

- The pre-refactor hand-authored templates
  (`infra/talos/patches/common.yaml.j2`, `controlplane.yaml.j2`) are gone.
- Replaced by render-owned **ordered, separate-file** patch layers under
  `infra/talos/patches/generated/{all,controlplane}/` (one file per
  document ‚Äî Talos forbids two patches touching the same document):
  - `all/00-install.yaml.j2` ‚Äî `machine.install.image` (resolved at
    bootstrap by `talos_schematic` from `talos_extensions` +
    `talos_version`, not at render time) + optional Talos API
    `certSANs`.
  - `controlplane/00-network.yaml.j2` ‚Äî private-network VIP via
    `network.interfaces` (with `deviceSelector` support, see
    Implementation notes).
  - `controlplane/10-cni-proxy.yaml.j2` ‚Äî `cluster.network.cni.name:
    none`, `cluster.proxy.disabled: true` (Cilium replaces both), plus
    optional kube-apiserver `certSANs`.
- All generated layer files start with `# Generated by render-config; do
  not edit.` and explain what they replace.

### Generated group_vars

- The pre-refactor hand-authored files
  `inventories/common/group_vars/all/{cluster,talos,flux}.yml` and
  `inventories/providers/hcloud/group_vars/all/hcloud.yml` are now
  **rendered** (headers in place). Variable names are preserved so
  existing roles (`talos_config`, `talos_schematic`, `hcloud_servers`,
  `talos_upgrade`) keep working unchanged.
- `hcloud.sops.yaml` stays user/secret-owned ‚Äî the render never decrypts
  it and never writes a token into a generated file.

### Flux catalog scaffolding (ADR-010, ADR-012)

- **New** `flux/infrastructure/controllers/_components/` (catalog, sparse
  today) with `providers/hcloud/` placeholder and a `README.md` that
  records the contract: each entry is an ordinary kustomize base
  (official Kustomize `kind: Component` is transform-only and cannot
  contain resources, so controller catalog entries are ordinary bases
  by default).
- **New** `flux/infrastructure/controllers/base/kustomization.yaml` ‚Äî
  render-owned, currently `resources: []` (sparse until feature work
  enables components).
- **New** `flux/infrastructure/controllers/base/cluster-vars.yaml` ‚Äî
  render-owned `ConfigMap` placeholder (`data: {}`) for future
  cluster-metadata consumers.
- `flux/infrastructure/controllers/kustomization.yaml` is the
  user-owned overlay that references `base/` and (optionally)
  `patches/`.
- `flux/infrastructure/controllers/patches/README.md` is the user-owned
  patches directory; empty for now.

  ‚Äî status changed to **Superseded by ADR-010** with a blockquote

- **`CLAUDE.md`** ‚Äî Implementation status updated to describe the

- **`docs/upgrades.md`**, **`docs/node-roles.md`**, **`docs/adding-a-provider.md`** ‚Äî

## CCM / `externalCloudProvider` status in this slice

Gated off. The configuration is wired through the compiler but the
default is off and no installer exists yet.

- `config/defaults/cluster.yaml` ships `features.hcloudCcm: false` with
  a comment warning that enabling it before the installer lands would
  taint nodes with no CCM to clear them ‚Äî so the gate is intentionally
  defensive, not aspirational.
- `config_render_cluster_external_cloud_provider_enabled` is computed
  in `roles/config_render/tasks/compute.yml` and gates the
  `all/10-cloud-provider.yaml.j2` Talos layer in
  `roles/config_render/tasks/render_talos.yml`. With the default, this
  layer is **not generated** into `infra/talos/patches/generated/all/`.
- The `hcloud-ccm` entry in `bootstrap_pre_flux_components` is also
  gated off; nothing is emitted into the pre-Flux bootstrap list today.
- **Next slice** (the revised `cilium-ccm-csi` plan) will:
  1. Add the hcloud CCM Helm release and `HelmRepository` to
     `flux/infrastructure/controllers/_components/providers/hcloud/ccm/`.
  2. Enable `features.hcloudCcm: true` in
     `config/overrides/cluster.yaml` (flipping the gate and the
     generated Talos layer together).
  3. Update `bootstrap_pre_flux_components` so CCM runs pre-Flux to
     clear the `uninitialized` taint, then is adopted by Flux from
     git.

## Implementation notes

- **Single source of truth for the VIP.** `cluster.vip` is the only
  place the VIP lives. The renderer feeds it into the Ansible
  `cluster_vip` var (used by the cluster endpoint and cert SANs) and
  into `controlplane/00-network.yaml.j2`. The schema explicitly
  rejects a parallel `talos.network.vip` key ‚Äî duplicating the VIP
  across two schema keys risks silent divergence.
- **`deviceSelector` support preserved but unused.** The full
  `talos.network.defaultInterface.deviceSelector` schema is wired
  through `compute.yml` and the `00-network.yaml.j2` template
  (`deviceSelector: <key>: <value>` branch with merge-on-selector
  semantics per Talos docs). With the current overrides the template
  emits the legacy `interface: eth1` form, so output is
  behavior-preserving. The plan's `talosctl` deviceSelector probe
  (run by the user after the cluster is bootstrapped) feeds
  `ansible-implementor` to wire a real selector in a later slice. The
  probe is **deferred** ‚Äî see Follow-ups.
- **Generated files are render-owned.** Every emitted file starts with
  `# Generated by render-config; do not edit.` and explains what
  manual file it replaces. Computed keys (final `talos_extensions`,
  `bootstrap_pre_flux_components`, the `externalCloudProvider` flag)
  are rejected if directly overridden ‚Äî keeps one writer per file.
- **Secret boundary preserved.** `hcloud.sops.yaml` is never read by
  the renderer and never written into a generated file. The hcloud
  inventory plugin still loads at inventory time (ADR-014); that
  boundary is unchanged from the prior slice.
- **No live cluster operations.** Per the plan, this slice ran no
  `hcloud`, `talosctl`, `kubectl`, `flux reconcile`, provisioning,
  bootstrap, or upgrade commands. The `infra/talos/secrets/` dir
  remains empty; no machine configs were rendered.
- **Kustomize terminology callout.** Controller catalog entries are
  ordinary kustomize bases, not official Kustomize `kind: Component`
  overlays. The `_components/README.md` records this so a future
  contributor doesn't reach for `kind: Component` expecting
  `resources:` to work.

## Follow-ups the user still needs to run

**None for this refactor slice.** The refactor is purely structural;
nothing is waiting on the user to push a config or restart a service.
The new flow is the same edit-override-render-push loop documented in
`CLAUDE.md` ("Push a config-only change"), which the user runs at
their own pace.

Items the user has *chosen* to defer (not blocking this slice):

- **`talosctl` deviceSelector probe** ‚Äî run after the cluster is
  bootstrapped (later feature slice):
  ```bash
  talosctl -n <node-ip> get links
  talosctl -n <node-ip> get hardwarecomponents
  ```
  Feed the output to `ansible-implementor` to derive a real
  `deviceSelector` and replace `interface: eth1` in
  `config/overrides/cluster.yaml`. If Hetzner virtualizes interfaces
  without stable PCI identities, document that and keep `interface:
  eth1` with a `# why` comment.
- **Revise `.ai/plans/2026-06-22-cilium-ccm-csi.md`** to consume the
  new compiler outputs and catalog layout before implementing
  feature work. The original plan was written before the refactor
  landed and assumes hand-authored Flux resources.
- **CI for render drift** ‚Äî not built; revisit when CI is introduced
  for the repo.
