# ADR-016 Talos configuration boundary migration — closing report

**Date:** 2026-06-25
**Plan:** `.ai/plans/2026-06-25-talos-config-boundary-migration.md`
**ADR:** `docs/cluster-bootstrap/adrs/016-talos-config-boundary.md` (Accepted; supersedes ADR-013)
**Type:** Refactor — no behavior change on rendered output, no live cluster operations

## What changed

The Talos configuration boundary was redrawn per ADR-016: human-authored Talos node configuration is now stored as **Talos-shaped machine config patch manifests** under `config/talos/base/` (template-owned, cluster-agnostic) and `config/clusters/kube1/talos/` (user-owned cluster overlay, with optional per-node `nodes/<hostname>.yaml`), plus `config/talos/components/` for feature/provider fragments selected by the render compiler (currently `cilium.yaml` and `external-cloud-provider.yaml`). The render compiler (`config_render` role) now composes a single ordered patch stack per Talos doc precedence: base → cluster overlay → components → computed integration fragments → per-node, with stable numeric prefixes (05/06/10/15/20/90) encoding the layering. The previous ADR-013 generated templates (`all/10-cloud-provider.yaml.j2`, `controlplane/10-cni-proxy.yaml.j2`) were split: their static content moved to `config/talos/components/`, and the formerly-bundled kube-apiserver certSANs became a new computed `controlplane/20-cert-sans.yaml.j2`. The `config/overrides/` tree is preserved as a **legacy fallback** — the loader prefers `config/clusters/kube1/{cluster,nodes}.yaml` (selected via `-e config_cluster=kube1`, the new default) and emits a deprecation warning when the fallback is used; `config/overrides/cluster.yaml` is now annotated "LEGACY FALLBACK — migration compatibility only". The `talos_config` consumer role was extended with worker-scope discovery (`generated/worker/`, `--config-patch-worker` argv), a `.gitkeep` sentinel is rendered when the worker directory is empty to keep check-mode diffs clean, and the `--config-patch-worker` flags are added to both the `talosctl gen config` argv and the regeneration SHA-256 signature. The generated Talos patch tree gained `worker/` and `nodes/` subdirectories; existing `all/00-install` and `controlplane/00-network` were renamed to `all/20-install` and `controlplane/15-network` so computed integration fragments apply **after** user overlays (later patches win per Talos strategic-merge). CLAUDE.md, `docs/upgrades.md`, `docs/node-roles.md`, and `docs/adding-a-provider.md` were updated to point at the new source paths, and ADR-013 was marked "Superseded by ADR-016". A guard was added to `render_talos.yml` that fails the render if a controlplane base/cluster overlay sets `machine.network.interfaces` (the VIP/interface merge-key hazard — Talos merges interfaces by name/deviceSelector, so a later user overlay would not "win" the way a naive author expects).

## What was validated

All local, read-only, no live cluster contact, no provider-costing commands, no SOPS decryption. **`ansible-lint`** (production profile) — PASS, 0 failures / 0 warnings across 62 files. **`ansible-playbook playbooks/render-config.yml -e config_cluster=kube1 -e config_render_check=true`** — PASS, "No render drift detected. Committed generated files match render output." (114 ok, 20 changed, 40 skipped, 0 failed; deterministic on re-runs). **`ansible-playbook -i inventories/common -i inventories/providers/hcloud playbooks/bootstrap-talos.yml --syntax-check`** — PASS. **`ansible-playbook -i inventories/common -i inventories/providers/manual playbooks/bootstrap-talos.yml --syntax-check`** — PASS. Render output spot-checks confirmed: the `cilium` and `external-cloud-provider` components are correctly selected (hcloud + cilium → both selected; manual → only `cilium`); the `all/10-external-cloud-provider.yaml.j2` generated patch sets `cluster.externalCloudProvider.enabled: true`; `controlplane/10-cilium.yaml.j2` carries `cluster.network.cni.name: none` + `cluster.proxy.disabled: true`; the per-node discovery loop is guarded by `config_render_talos_node_names` so the new overlay path and the legacy `nodes[].patch` fallback don't both render for the same node; the `bootstrap_pre_flux_components` and `config_render_talos_components` lists stay in lock-step with the previous slice's contract.

## What was explicitly not run

Per the plan's safety boundary, agents did **not** run: `bootstrap-talos.yml` without `--syntax-check`, `bootstrap-flux.yml`, `provision-infra.yml`, `image-talos.yml`, `upgrade-talos.yml`, any `hcloud` API call, any `talosctl apply-config` / `talosctl upgrade` / `talosctl patch machineconfig`, any `kubectl` against a live cluster, any `flux reconcile`, any local `talosctl gen config` validation against rendered outputs (the renderer only writes `.yaml.j2` templates; the secrets dir remains untouched). The `infra/talos/secrets/` directory is empty. The hcloud `server.datacenter` deprecation warning on syntax-check is pre-existing (the hcloud CCM chart was pinned to 1.33.0 in the prior slice to avoid that) and not introduced by this migration. The "merged" `config/overrides/cluster.yaml` was kept verbatim as a legacy fallback — no functional drift; the only edits to it are the leading comment header.

## Known follow-ups / risks

- **The `talosctl` deviceSelector probe is still deferred.** `config/clusters/kube1/cluster.yaml` still sets `talos.network.defaultInterface.interface: eth1` (with `deviceSelector: null`). The controlplane Talos overlay is empty by design (comment-only) — VIP and interface selection remain in the computed `15-network.yaml.j2` integration fragment sourced from `cluster.vip` and `talos.network.defaultInterface`, avoiding VIP duplication across human Talos YAML and the schema. Migrating interface selection to a real `deviceSelector` in `config/clusters/kube1/talos/controlplane.yaml` is staged behind the live probe (the merge-key hazard guard added in this slice will catch any user overlay that accidentally re-introduces the same `machine.network.interfaces` item with a different key).
- **Legacy `config/overrides/` is still on disk.** Both legacy files are now marked "LEGACY FALLBACK"; the renderer emits a clear deprecation warning if a user removes the selected-cluster files and falls through. Removing the fallback is a separate cleanup slice after the team confirms no external automation depends on the old paths.
- **`features.cni` / `features.hcloudCcm` / `talos.network.vip` / `talos.install.disk*` / `storage.volumes` are still in `config/schemas/cluster-input.schema.json` and the `talos.*` raw-fragment fields (`talos.patches.{all,controlplane}`, `talos.install`, `talos.network.defaultInterface`) are still wired through.** These are all rejected at validate time when set to non-default values (the pre-existing transitional-key asserts), but the schema surface is not yet narrowed. A schema-narrowing cleanup slice is the natural follow-up once the team is comfortable with the new layout.
- **Live cluster state is unverified.** As with prior slices, "Working" in the CLAUDE.md status block means the code is built and passes review/validation — not that `bootstrap-flux.yml` has been observed running on a live cluster. The same live-validation commands from the prior slice's report apply (Cilium `daemonset/cilium` Ready, no `uninitialized` taint, every `providerID` starts with `hcloud://`, `flux-system` pods Ready, all HelmReleases Ready without install/uninstall churn).
- **Worker-scope patch discovery is wired but unused on kube1** (3 hybrid nodes only, no pure workers). The render emits a `worker/.gitkeep` sentinel to keep the directory commit-stable; the infrastructure exists so future pure-worker nodes added to the topology pick up `config/talos/base/worker.yaml` + `config/clusters/kube1/talos/worker.yaml` without role-code changes.
- **The ADR-017 Flux shared-overlay-ownership migration is in flight in parallel** (see `.ai/plans/2026-06-25-flux-shared-overlay-ownership-migration.md` and the new `flux/infrastructure/controllers/generated/selected/`, `flux/infrastructure/{configs,apps}/{base,overlays}/`, `flux/apps/{base,overlays}/` directories). That work overlaps with this slice on the Flux base kustomization renderer but is not part of the ADR-016 boundary; expect a follow-up closing report.
