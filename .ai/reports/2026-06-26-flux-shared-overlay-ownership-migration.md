# Flux shared overlay ownership migration — closing report

**Date:** 2026-06-26
**Plan:** `.ai/plans/2026-06-25-flux-shared-overlay-ownership-migration.md`
**ADR:** `docs/cluster-bootstrap/adrs/017-shared-overlay-ownership.md` (Accepted)
**Type:** Refactor — no behavior change on rendered output, no live cluster operations

## What changed

kube1's Flux tree was migrated from the ADR-010/011/012 layout (`catalog + render-owned base + single user overlay`) to the ADR-017 four-tier overlay ownership model across all three Flux domains. The new layout is:

- `_components/` — TEMPLATE-OWNED catalog (Cilium, hcloud CCM; unchanged name)
- `generated/selected/` — GENERATED by `config_render` (replaces render-owned `base/`)
- `overlays/common/` — REPO-WIDE user patches
- `overlays/clusters/kube1/` — PER-CLUSTER user patches

The three top-level kustomizations (`flux/infrastructure/controllers/kustomization.yaml`, `flux/infrastructure/configs/kustomization.yaml`, `flux/apps/kustomization.yaml`) were switched to shim onto `overlays/clusters/kube1`, so `flux/clusters/kube1/*.yaml` cluster entrypoint paths are unchanged. The cluster entrypoint's `Kustomization.spec.path` values stay exactly the same — the shim strategy avoids a `prune` blast radius from a `Kustomization.spec.path` change.

The render compiler (`config_render` role) was updated: a new `config_render_output_flux_selected` var in `defaults/main.yml` records the generated path; `render_flux.yml` now writes `flux/infrastructure/controllers/generated/selected/kustomization.yaml` from a new `templates/flux/selected/kustomization.yaml.j2`; resource paths in the template use `../../_components/` (the rendered file sits one level deeper than the previous `base/`); `check.yml` diffs the new selected path. The deprecated `flux/infrastructure/controllers/base/kustomization.yaml`, the orphaned `controllers/base/cluster-vars.yaml`, the stale `controllers/patches/README.md` and its empty `patches/` directory were removed. Ownership comments were added to every Flux kustomization file (TEMPLATE-OWNED / GENERATED / REPO-WIDE / PER-CLUSTER) so the boundary is explicit at the file level. `CLAUDE.md` (Repository structure, Implementation status, Flux specifics) and ADR-010's implementation-status note were updated to reflect the new layout. ADR-012 was *not* amended — implementation stayed inside ADR-017's accepted model.

All four new per-cluster overlay chains build (`controllers/overlays/clusters/kube1`, `controllers/overlays/common`, `configs/overlays/clusters/kube1`, `apps/overlays/clusters/kube1`). ## What was explicitly not run

Per the plan's safety boundary, agents did **not** run: `bootstrap-flux.yml`, `bootstrap-talos.yml`, `provision-infra.yml`, `image-talos.yml`, `upgrade-talos.yml`, any `hcloud` API call, any `talosctl apply-config` / `talosctl upgrade` / `talosctl patch machineconfig`, any `kubectl` against a live cluster, any `flux reconcile`, any `helm` operations against a live cluster. Local build commands only (`kustomize build`, `ansible-playbook --check`, `ansible-lint`); no live-cluster validation was performed. Empty placeholder `overlays/user/` directories exist on disk from an earlier plan iteration and will disappear at commit time (git does not track empty directories) — no action required.

## Known follow-ups / risks

- **`_components/` rename to `components/` is still deferred.** ADR-017's illustrative target uses `components/`. The current plan kept `_components/` to minimize path churn during the migration. This is a small, isolated follow-up — touching the catalog directory plus the resource paths in `render_flux.yml` + the selected template. A separate closure report would be overkill.
- **No live-cluster validation.** The kustomize builds, render check, and lints all pass locally. The first live verification must be done by the user: `flux reconcile kustomization flux-system --with-source` and `flux get kustomizations -A` to confirm the three infra-controllers / infra-configs / apps Kustomizations still report `Ready` after the path change. If the migration produced any local-only path mistake, Flux garbage collection could prune an unexpectedly missing resource — the cluster entrypoint shim strategy minimizes this risk (Kustomization paths are unchanged), but the first reconcile is the real proof.
- **Three of the new overlay directories are currently empty by design.** `flux/infrastructure/controllers/overlays/common/`, `flux/infrastructure/configs/{base,overlays/common,overlays/clusters/kube1}/`, `flux/apps/{base,overlays/common,overlays/clusters/kube1}/` all contain only a `kustomization.yaml` referencing the next layer down. They will become useful the moment a cluster or a repo-wide patch is added; they exist now to enforce the ownership model before content lands.
- **ADR-012 implementation-status note still references the old two-tier model in places outside the bullet that was updated.** A separate docs cleanup pass is warranted if the old prose is read in isolation — `kustomize` introspection on a per-file basis would be needed to enumerate remaining stale references. Not blocking.
- **Pre-Flux adoption contract is preserved but not re-validated live.** The plan required that release names, namespaces, chart versions, and values stay equivalent; the local builds confirm this. A live `helm list -n cilium-system` and `helm list -n hcloud-ccm-system` cross-check after the next `flux reconcile` is the final verification (the same `helm` cross-check that the pre-Flux bootstrap section already calls for).
